# DireEngine Cleric Canon Audit

Generated by DRG-CANON-AUDIT-002 on 2026-05-14. This document classifies all existing Cleric-related code paths by canon provenance and recommends policy per system. It is the authoritative reference for canon decisions during the DRG-CLERIC program.

## Summary

- (a) 2004-GSL canon: 0 systems
- (b) Magic 3.0 / Elanthipedia: 3 systems
- (c) DireEngine-bespoke: 7 systems
- (d) Provenance unknown: 0 systems
- (e) GSL-mismatched: 1 system

## Top-level policy decision required

Decide whether the existing devotion-plus-favor Cleric/death loop is a grandfathered DireEngine class identity or a temporary pre-canon scaffold that will be replaced by SAF, deity alignment, and S00963 ritual staging.

That decision is high-impact because the current model is not isolated inside Cleric-only code. `devotion` and `favor` already affect generic mana services, death services, UI payloads, commands, DireTest scenarios, and smoke tooling.

## Per-system audit

### System 1: Devotion attribute

- **Provenance:** (b) Magic 3.0 / Elanthipedia origin
- **Evidence:** `CLERIC_DEVOTION_CONFIG`, `get_devotion()`, `set_devotion()`, and `adjust_devotion()` define devotion as a persistent Cleric-only resource in [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L933), [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L3254), and [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L3278). The archived design note explicitly describes devotion as a Cleric resource sustained by rituals, spent by communes, and affecting spell power, with Elanthipedia cited as authority in [docs/archive/root-workfiles/Cleric.md](c:/Users/gary/dragonsire/docs/archive/root-workfiles/Cleric.md#L11).
- **Mechanical role:** Devotion is a Cleric-only spiritual resource with a max, gain/loss rules, state labels, shrine regen, spell-power scaling, commune costs, corpse-rite costs, and resurrection quality/failure modifiers.
- **GSL comparison:** The established GSL anchor for Cleric piety is S00963 SAF/uncleanliness staging, not a percentage-based devotion meter that directly increases holy power. GSL-style devotion language may exist around Cleric flavor, but this implementation is not the S00963 control seam.
- **Recommended policy:** `hybrid`
- **If replace:** Replace devotion-as-primary-canon with SAF for canonical piety gating, remove devotion-based holy scaling, and re-scope current devotion calls as either deleted behavior or a narrower DireEngine-only resource.
- **If grandfather:** Tag devotion explicitly as `DireEngine grandfathered Cleric spirituality layer` in docs and code comments, and stop treating it as implied DR 2004 canon.
- **If hybrid:** Keep devotion as a secondary class resource for commune/resurrection tuning, but make SAF the authoritative canon piety gate for ritual state and spell permission.

### System 2: Favor attribute

- **Provenance:** (b) Magic 3.0 / Elanthipedia origin
- **Evidence:** Favor is implemented as a separate persistent resource in [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L2801), [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L2895), and [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L3186). The archived Cleric design note explicitly distinguishes favor from devotion and frames it as a player-wide death-protection currency sourced from Elanthipedia-era DR understanding in [docs/archive/root-workfiles/Cleric.md](c:/Users/gary/dragonsire/docs/archive/root-workfiles/Cleric.md#L11).
- **Mechanical role:** Favor is a player-wide death/recovery resource used for depart, corpse snapshots, resurrection viability, and death-sting mitigation, with prayer gain and passive decay.
- **GSL comparison:** The established 2004-GSL Cleric anchors in this project center on SAF, profession 3, Holy realm 2, S00465, and S00830. Favor as a distinct top-level death currency is not part of that current GSL anchor set.
- **Recommended policy:** `grandfather`
- **If replace:** Replacing favor would force a broad redesign of depart, corpse snapshots, resurrection viability, and death UX across non-Cleric systems.
- **If grandfather:** Tag favor explicitly as `DireEngine/Elanthipedia-derived death resource`, keep it out of the canon Cleric spell/ritual model, and treat it as world-death infrastructure rather than Cleric canon.
- **If hybrid:** Retain favor for death/depart scaffolding while preventing it from being mistaken for Cleric piety or deity standing.

### System 3: Shrine mechanics

- **Provenance:** (c) DireEngine-bespoke
- **Evidence:** Rooms expose a generic `db.is_shrine` / `is_shrine_room()` seam in [typeclasses/rooms.py](c:/Users/gary/dragonsire/typeclasses/rooms.py#L76) and [typeclasses/rooms.py](c:/Users/gary/dragonsire/typeclasses/rooms.py#L183). `pray_at_shrine()` grants timed favor and devotion regeneration with timestamp throttles in [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L3204).
- **Mechanical role:** Shrines are location flags that reset death streak state and act as passive or semi-passive favor/devotion refill points.
- **GSL comparison:** Shrines are canonically meaningful ritual spaces, but the current mechanic is a timed regeneration pad. That is not the same as S00963 ritual reward flow.
- **Recommended policy:** `replace`
- **If replace:** Shrines should become canonical ritual/deity interaction sites, not passive devotion/favor batteries.
- **If grandfather:** Only grandfather the room-tagging seam, not the current regen behavior.
- **If hybrid:** Keep `is_shrine` as a room capability flag, but replace shrine regen with ritual/deity interactions and move any remaining regen to a clearly non-canon support layer.

### System 4: Commune mechanics

- **Provenance:** (b) Magic 3.0 / Elanthipedia origin
- **Evidence:** `CmdCommune` routes `commune` to `commune_with_divine()` in [commands/cmd_commune.py](c:/Users/gary/dragonsire/commands/cmd_commune.py#L1). `commune_with_divine()` currently supports `solace`, `ward`, and `vigil` as active abilities in [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L4017), and the archived Cleric note describes that exact loop in [docs/archive/root-workfiles/Cleric.md](c:/Users/gary/dragonsire/docs/archive/root-workfiles/Cleric.md#L1).
- **Mechanical role:** Commune is currently an active spell-adjacent verb that spends devotion to restore attunement, apply a ward, or stabilize a corpse.
- **GSL comparison:** In the scoped canon plan, commune is tied to deity alignment and religious state, not a three-ability resource-spend menu. The current implementation uses a DR-flavored word for a different gameplay surface.
- **Recommended policy:** `replace`
- **If replace:** Reserve `commune` for deity alignment/guidance and migrate the current `solace` / `ward` / `vigil` effects to a different verb or explicitly grandfathered sub-system.
- **If grandfather:** Tag the current commune set as post-2004/Elanthipedia-inspired and rename it in docs as a non-canon exception.
- **If hybrid:** Keep the effect implementations, but move them off the canonical `commune` verb so deity-facing canon can use the term cleanly.

### System 5: Cleric ritual logic

- **Provenance:** (c) DireEngine-bespoke
- **Evidence:** General prayer rituals are defined through `CLERIC_DEVOTION_CONFIG["rituals"]`, `get_cleric_ritual_profile()`, and `perform_cleric_ritual()` in [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L933) and [typeclasses/characters.py](c:/Users/gary/dragonsire/typeclasses/characters.py#L3938). - **Mechanical role:** The current ritual layer is two systems: short devotion rituals through `pray`, and multi-stage corpse rites (`prepare`, `stabilize`, `restore`, `bind`) that drive resurrection setup.
- **GSL comparison:** This is not S00963's stage-one/stage-two/stage-three SAF ritual engine. It is a custom death-and-devotion ritual architecture using Cleric terms.
- **Recommended policy:** `replace`
- **If replace:** Build the canonical SAF ritual engine separately and migrate corpse-rite behavior into a different subsystem name or fold only the reusable scheduler/message infrastructure forward.
- **If grandfather:** Grandfather only the corpse-rite chain as DireEngine death gameplay, not as canonical Cleric ritual implementation.
- **If hybrid:** Keep delayed stage completion, participant tracking, and messaging helpers, but replace ritual identity, rewards, and piety semantics.

## Highest-impact recommendations

- Treat devotion, commune, shrine regen, and current ritual handling as one connected design family. Do not make piecemeal canon changes to one of those surfaces without deciding the policy for all four.
- Grandfather favor and the broader death/recovery loop unless there is explicit appetite for a much larger death-system rewrite. Favor is already a world system, not just a Cleric mechanic.
- Reclassify the Cleric-facing spell registry before any further Cleric dispatches. The current default `gsl_2004` provenance on non-canonical rows is the audit's clearest `GSL-mismatched` finding.
- Keep the built Cleric guildhall and world scaffolding, but replace the learning/progression semantics with canon-aware guild behavior instead of rebuilding the area from scratch.
- Split future Cleric work into `canon class identity and piety` versus `DireEngine death-support mechanics`. The repo currently conflates them.

## Open questions

- Should devotion survive as a grandfathered secondary Cleric resource after SAF is introduced, or should SAF fully replace it as the sole piety mechanic?
- Should `commune` be reclaimed for deity alignment/guidance, with current `solace` / `ward` / `vigil` moved elsewhere?
- Should the resurrection/corpse system be treated as project-canonical death infrastructure with Cleric hooks, or does Gary want the Cleric program to normalize it closer to a narrower DR spell-and-ritual model?
- How aggressively should existing tests be rewritten once provenance policy is decided? Several current tests are preserving exactly the behavior this audit classifies as non-canon or bespoke.

## Policy decisions under DRG-CANON-POLICY-001

DRG-CANON-POLICY-001 resolves the top-level policy question for Cleric in favor of hybrid-leaning-grandfather. The working default for this class is:

- existing functioning Cleric systems are grandfathered as project-canonical and tagged `directengine_canon`
- missing canonical 2004-GSL content is added where the class lacks it entirely and tagged `gsl_2004`
- deliberate post-DR3 presentation plus 2004-GSL mechanical bridges are tagged `hybrid_design`

This section is prescriptive. Future Cleric dispatches should follow these decisions unless a later policy dispatch explicitly changes them.

### System 1: Devotion attribute

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** Devotion is a substantial, working Cleric-only class resource already integrated into commune, resurrection, holy scaling, commands, UI, and tests. Under the new policy, it is not replaced by SAF just because 2004-GSL used a different piety seam.
- **Future add rule:** If any missing canonical Cleric content needs piety gating, adapt it around the grandfathered devotion model unless a later class-boundary review changes policy.

### System 2: Favor attribute

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** Favor is already live world-death infrastructure, not merely a tentative Cleric experiment. Replacing it would exceed Cleric scope and violate the default against rebuilding working systems.
- **Future add rule:** New Cleric content may depend on existing favor semantics where appropriate, but favor itself is not reopened during the Cleric program.

### System 3: Shrine mechanics

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** The current shrine flag and shrine prayer loop are working Cleric/death support mechanics. The hybrid-leaning-grandfather default keeps them live rather than replacing them with a new SAF ritual-site engine.
- **Future add rule:** If canonical shrine-facing content is added later, it should extend this seam rather than deleting the existing one unless a specific conflict is documented.

### System 4: Commune mechanics

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** `solace`, `ward`, and `vigil` already define a working Cleric class verb surface. Under this policy they stay canonical for DireEngine Cleric rather than being displaced by a deity-alignment rewrite.
- **Future add rule:** Additional commune-facing content should extend the current model unless a new dispatch documents a concrete conflict that justifies a hybrid or replacement decision.

### System 5: Cleric ritual logic

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** The current ritual layer, including short prayer rites and corpse-stage rites, is working gameplay with meaningful class identity. The policy default is to keep it.
- **Future add rule:** Missing canonical ritual flavor or progression may be added around this system, but the existing ritual engine is not a replacement target in the Cleric scope rewrite.

### System 6: Resurrection / corpse mechanics

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** The resurrection and corpse system is a large, validated DireEngine gameplay surface with cross-system dependencies. It remains project-canonical for Cleric work.
- **Future add rule:** Canonical Cleric additions may enrich this system, but the underlying death pipeline is not reopened by default.

### System 7: Cleric guildhall

- **Policy decision:** Grandfather existing world scaffolding as `directengine_canon`; add missing canonical guild mechanics as `gsl_2004` where absent.
- **Why:** The built guildhall, rooms, and guildmaster presence already work and should remain. The gap is missing canonical interaction depth, not missing world scaffolding.
- **Future add rule:** Guild-guy learn-spell flow, circle checks, or other absent canonical guild interactions can be added as `gsl_2004` without rebuilding the area.

### System 8: Cleric spell registry

- **Policy decision:** Add missing canonical Cleric spellbook content as `gsl_2004`; grandfather existing working non-canonical Cleric-facing rows as `directengine_canon` until individually reclassified; use `hybrid_design` only for deliberate bridge cases.
- **Why:** This is the audit's clearest mismatch. The policy does not justify deleting the current live registry wholesale, but it does require stopping the assumption that missing Cleric spellbook content is already covered.
- **Future add rule:**
	- missing 206xxx Cleric spell content is added as `gsl_2004`
	- existing working rows that are not canonical 2004 Cleric spells are not treated as `gsl_2004` by default
	- any deliberate post-DR3 presentation with 2004-GSL math is tagged `hybrid_design`

### System 9: Cleric profession identity

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** String-keyed profession identity is an implementation representation choice, not a gameplay defect. It already works across class gates, guild tags, realm routing, and UI.
- **Future add rule:** Canonical metadata can be layered onto this identity model without replacing the storage form.

### System 10: Holy mana realm routing

- **Policy decision:** Grandfather as `directengine_canon`.
- **Why:** Holy realm routing already works and is a stable Cleric seam. The policy does not reopen it simply because the exact representation differs from raw GSL internals.
- **Future add rule:** New canonical Cleric spell content should use the existing Holy routing. Any deliberate bridging of presentation and 2004 mechanical math can be tagged `hybrid_design` if needed.

### System 11: Tests for existing Cleric content

- **Policy decision:** Grandfather existing behavior-preservation tests as `directengine_canon`; add new canonical coverage as `gsl_2004` where new canonical content is introduced.
- **Why:** The current suite guards live Cleric behavior that this policy now treats as project-canonical. Those tests should not be discarded simply because the original audit found non-2004 provenance.
- **Future add rule:** The Cleric program should split legacy-preservation coverage from new canonical spellbook/guild coverage so both policies remain visible.

## Cleric scope rewrite implications

Under DRG-CANON-POLICY-001, the next Cleric roadmap rewrite should assume:

- devotion, favor, shrine, commune, ritual, resurrection, profession identity, and Holy routing remain in scope as grandfathered `directengine_canon`
- guild depth and missing spellbook content are the main `gsl_2004` additions
- deliberate bridge content remains available through `hybrid_design`, but only by explicit choice

This resolves the prior audit's open policy question for the Cleric class.
