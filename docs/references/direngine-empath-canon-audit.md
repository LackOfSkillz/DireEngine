# DireEngine Empath Canon Audit

Generated by DRG-EMPATH-AUDIT on 2026-05-15. This document classifies the current Empath implementation under the post-Cleric `B` policy posture and defines what must happen before the Empath program can be treated as settled.

## Summary

- Profession identity exists and is usable today: Empath is a string-keyed profession, routes to the Life mana realm, has a dedicated guildhall, and retains a custom Empath-only circle path.
- Class-identity mechanic classification: **II**.
- Wound-transfer exists, is command-driven, and was verified live on `SmokeEmpathLive`, but the structured spell/guild progression layer does not line up with that identity cleanly enough to call the grandfather settled.
- Empath guild infrastructure exists in-world, but `EmpathGuildleader.teach_spell(...)` is currently miswired and refuses even current Empath spell rows.
- The canonical 401-404 Empath book is mostly absent from the structured spell registry. The registry currently carries two Empath-specific rows (`Heal`, `Regenerate`) plus a large shared/cross-profession roster that is not class-specific Empath content.

## Top-level policy outcome

Empath does **not** halt under the DRG-EMPATH-AUDIT stop condition.

The class-identity loop is present, so this is not Class III. But it also is not ready for a straight Cleric-pattern content program. The controlling result of this audit is:

- Keep hybrid-leaning-grandfather as the default posture for Empath.
- Do **not** treat the current Empath grandfather as settled yet.
- Insert **DRG-EMPATH-FOUNDATION-001** before the normal Empath content sequence to reconcile wound-transfer, spell identity, and guild progression.

That makes Empath a **Path B** class under the audit rubric: the core identity exists, but the current implementation needs a bounded refinement pass before standard content dispatches.

## Section B: Class-Identity Mechanic — Wound Transfer

This is the controlling section for the audit.

### B1. Existence check

- **Finding:** wound-transfer exists directly in the runtime.
- **Core implementation:** `typeclasses/characters.py::transfer_wounds(self, target)` plus the larger Empath transfer path in `take_empath_wound(...)`.
- **Command surfaces found:** `touch`, `link`, `take`, `heal`, `mend`, `purge`, `stabilize`, `perceive`, `manipulate`, `unity`, `redirect`.

This is not a paper design or stub. The class-identity mechanic is implemented.

### B2. Current command surface

- `touch <target>` establishes a diagnostic link.
- `link <target>` deepens the empathic bond.
- `take <wound> [amount]` draws a linked patient's condition into the Empath.
- `heal wounds` and `heal vitality` route into the same transfer family through `take_empath_wound(...)`.
- `mend self` reduces the Empath's carried vitality / bleeding burden after transfer.

### B3. Current state transition

The current live loop is:

1. establish link to patient
2. draw wound or vitality through `take`
3. patient burden decreases
4. Empath burden rises
5. Empath stabilizes themselves with `mend self` or related tools

This is recognizably the correct Empath gameplay identity.

### B4. Live verification

Live verification was executed in `/play` on `SmokeEmpathLive`.

Setup:
- moved `SmokeEmpathLive` into `Empath Guildleader's Office`
- created a controlled target `EmpathAuditPatient` in the same room
- set target wound state via in-band `py`

Baseline readback:
- `self_before`: `{'fatigue': 0, 'vitality': 10, 'bleeding': 2, 'poison': 0, 'disease': 0}`
- `patient_before`: `{'fatigue': 0, 'vitality': 12, 'bleeding': 18, 'poison': 0, 'disease': 0}`

Commands issued:
- `touch EmpathAuditPatient`
- `link EmpathAuditPatient`
- `take bleeding 10`
- `mend self`

Observed live command text:
- `You reach out and sense the condition of your patient.`
- `You deepen your connection, taking in the shape of their suffering.`
- `You draw the injury into yourself.`
- `You focus inward, stabilizing your condition.`

Readback after `take bleeding 10`:
- `self_after_take`: `{'fatigue': 5, 'vitality': 10, 'bleeding': 14, 'poison': 0, 'disease': 0}`
- `patient_after_take`: `{'fatigue': 0, 'vitality': 14, 'bleeding': 8, 'poison': 0, 'disease': 0}`

Readback after `mend self`:
- `self_after_mend`: `{'fatigue': 5, 'vitality': 0, 'bleeding': 9, 'poison': 0, 'disease': 0}`
- `patient_after_mend`: `{'fatigue': 0, 'vitality': 15, 'bleeding': 8, 'poison': 0, 'disease': 0}`

Interpretation:
- the wound-transfer loop is live
- it is command-driven, not merely hidden in tests
- the patient gets relief and the Empath absorbs burden
- the Empath then stabilizes themselves

### B5. Comparison against the current structured spell layer

- `domain/spells/spell_definitions.py` currently exposes `SPELL_REGISTRY["empath_heal"]` as `Heal` and `SPELL_REGISTRY["regenerate"]`.
- `engine/services/spell_effect_service.py::_apply_healing_spell(...)` treats `empath_heal` as shared HP restoration routed through `StateService.apply_healing(...)`.
- That structured spell path heals hit points directly. It does **not** model the wound-transfer loop just verified live.

This is the key mismatch.

### B6. Canon comparison

Under the audit brief's canonical inventory, the controlling healing book includes:
- Vitality Healing
- Heal Wounds
- Heal Scars
- Heal
- Regenerate

The current live wound-transfer loop is closer to canonical Empath identity than the current structured `empath_heal` row is. But the spell registry and command/runtime seams are not aligned enough to call the grandfather canon-settled.

### B7. Equivalent of devotion/favor

- No Cleric-style devotion/favor twin exists for Empath.
- The main class-specific internal resources/mechanics are instead wound load, transfer shock, overload, link stability, unity/channel states, and life-force perception.
- `Transfer: 10/10` displayed live in the browser header for `SmokeEmpathLive`.

This is a different class identity shape, not a missing Cleric-style piety resource.

### Section B classification

**Classification: II**

Why:
- the wound-transfer mechanic exists
- it works live
- but the structured spell layer, guild teaching layer, and canonical spell naming are not yet aligned with that identity

### Section B required consequence

Add **DRG-EMPATH-FOUNDATION-001** before normal content dispatches.

That dispatch should:
- decide and document the authoritative relation between the command-driven transfer loop and the structured healing spell rows
- repair the Empath guild teaching seam so Merla can actually teach Empath spells
- settle the naming/provenance relationship between current `Heal`, command-level wound transfer, and the canonical 401 healing book

## Audit conclusion

Empath is a **Class II** audit result.

The class identity is real and already feels like Empath gameplay in live use. That is the strongest result of the audit.

But the program cannot skip straight to the standard Cleric-like spellbook rollout because three seams are misaligned:

1. the wound-transfer identity loop lives in direct Empath commands and character methods, not in a clean canonical healing-book spell model
2. the current structured `Heal` spell row does not capture the identity that the live class actually uses
3. Merla's guild teaching seam is currently miswired and refuses even present Empath spells

Therefore the correct next step is **DRG-EMPATH-FOUNDATION-001**, followed by the normal Empath sequence.
