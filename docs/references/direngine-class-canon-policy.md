# DireEngine Class Canon Policy

Generated by DRG-CANON-POLICY-001 on 2026-05-14. This document defines how DireEngine class implementations are treated as canon during Cleric, Empath, Ranger, Thief, and future class work.

## Policy

DireEngine class implementations follow **hybrid-leaning-grandfather** canon authority.

### Default rule

Existing DireEngine implementations are project-canonical. Grandfather them. Document provenance. Extend with canonical 2004-GSL mechanical depth where the existing system has clear gaps. Do not replace working DireEngine code with canonical-2004 code unless there is a specific reason articulated and approved.

The default starts class-by-class. Cleric grandfathers first because:

- the existing implementation is substantial and works
- the speed gain matters for the 16-day window
- DireEngine's design pattern of "3.0 names on 2004 engine" is already established by the four grandfathered Magic 3.0 spells

### Revisit checkpoint after each class ships

When Cleric completes, evaluate whether the grandfather decision produced gameplay we are happy with. If yes, apply the same default to Empath. If no, adjust the policy for Empath and document why. Apply the same revisit step per Ranger, Thief, and later classes.

### This is not pure grandfather

Where the existing implementation is *missing* canonical content with no conflicting design, the canonical 2004-GSL content gets added. The hybrid lean means:

- grandfather what conflicts with canon but already works
- add what is absent from canon
- document every deliberate mix

## Provenance taxonomy

Every class-related code path gets one of three provenance tags:

### `directengine_canon`

Grandfathered as project-canonical. Existed before this policy was set and is declared canonical by this policy or a per-class audit decision. Mechanics stay frozen unless a later policy change explicitly reopens them.

### `gsl_2004`

Sourced from 2004-GSL scripts. Mathematics and mechanics match canonical 2004 source.

### `hybrid_design`

Uses 2004-GSL mechanics under post-DR3 presentation, naming, or packaging. The existing pattern example is the approved four-spell Magic 3.0 exception set. New hybrid content gets this tag only when the mix is deliberate and documented.

Each existing class system gets its tag during the class's canon audit. New content gets its tag when created.

## Decision shortcuts

To prevent every dispatch from re-litigating canon questions, use these shortcuts.

### 1. Working DireEngine class system exists

If a working DireEngine system already exists for a class concept, default to grandfather it. Tag it `directengine_canon`. Do not rebuild it just because 2004-GSL did something differently. Audit more deeply only if a specific conflict surfaces during implementation.

### 2. GSL has content the class lacks entirely

If GSL has content the class lacks entirely, add it as `gsl_2004`.

Examples:

- canonical 206xxx Cleric spells not yet implemented
- S00465 guild-guy mechanics that do not exist yet

Adding where absent does not require a separate policy debate.

### 3. Conscious mix wanted

If a design deliberately mixes post-DR3 presentation with 2004-GSL mechanics, tag it `hybrid_design` and document the mix. This is used sparingly and is never the default.

## What this policy is not

### Not a permanent freeze

Project-canonical means treated as canon for now and revisited per class boundary. If Cleric's grandfathered model produces bad gameplay, Empath may get a different policy.

### Not an excuse to skip thinking

Defaulting to grandfather does not eliminate judgment. When a grandfathered design conflicts directly with desired new content, surface the conflict in the dispatch and resolve it deliberately.

### Not a downgrade of canon authority discipline

This policy adds class-implementation rules on top of DRG-CANON-AUDIT-001 discipline. It does not override those broader rules.

The standing authority rules remain:

- 2004-GSL is still the canonical mechanical source where DireEngine has no existing implementation
- Elanthipedia is still not canon authority
- Magic 3.0 grandfathering still requires explicit approval
- the four grandfathered Magic 3.0 spells remain the only already-approved Magic 3.0 exception set unless a later policy says otherwise

## Effect on Empath, Ranger, and Thief

Each class gets its own canon audit before its scope is drafted. Each audit applies this same policy default:

- grandfather what already works
- add what is absent from GSL where needed
- document deliberate hybrids

The revisit checkpoint still applies. Cleric's outcome informs Empath, but does not lock Empath, Ranger, or Thief into the same answer if their audits show different reality.

## Operating rule for future dispatches

When a future class dispatch touches an already-working class system, the default question is no longer "how do we replace this with 2004-GSL?" The default question is "is this working system being grandfathered as `directengine_canon`, and if not, what specific conflict justifies reopening it?"

## Policy revisit checkpoint — Cleric program

Date: 2026-05-15
Decision: B

Rationale:

The hybrid-leaning-grandfather policy produced acceptable Cleric gameplay, but the closeout smoke showed where future class work needs tighter discipline. Cleric is now playable as a coherent class: Esuin teaches the shipped Cleric spellbook with live circle gating, Cleric `circle` uses explicit Primary Magic plus Theurgy requirements, shrine prayer and favor work in-band, resurrection enforces consecrated-space rules, and all five shipped spell sub-books reached live smoke without a traceback. The missing SAF seam did not undermine overall class play; it only kept Huldah's Pall in the documented deferred queue. The held-mana bundle also behaved honestly at closeout: `Mass Rejuvenation` was learnable through the live guildmaster path and then returned a clear placeholder message instead of failing opaquely.

The caveat is that future class programs still need command-level and smoke-level proof rather than relying on structural similarity alone. Cleric closeout surfaced two important discipline lessons: first, live command-boundary checks are essential because they proved Cleric correctly refused Empath spells at the user-facing `learn ... from ...` surface; second, signature class mechanics should not be assumed safe just because the grandfathered system is broadly functional. Cleric's grandfathered devotion/favor/shrine model worked well, but Empath's signature wound-transfer loop is central enough that it deserves the same style of end-to-end smoke and boundary verification before its policy posture is treated as settled.

Effect on Empath program:

Empath should keep the same default policy posture as Cleric:
- grandfather working DireEngine class-identity systems by default
- add missing canonical 2004-GSL content where absent
- document hybrids deliberately

But Empath planning should add two mandatory checks up front:
- explicit command-level boundary verification against Cleric and other off-class spell surfaces
- explicit live smoke of Empath's signature wound-transfer and self-healing loop before treating the grandfathered mechanic as conclusively acceptable

Specific learnings from Cleric grandfather:
- What worked:
	- devotion, favor, shrine prayer, and resurrection all integrated cleanly with the canonical Cleric spellbook
	- canonical 206xxx spells fit the grandfathered class frame without forcing a redesign
	- per-class guild progression could be layered cleanly on top of shared infrastructure
- What didn't:
	- assumptions made from design notes can drift from shipped registry data, as seen with Aesrela Everild being learnable at circle 10 while Halo remained the actual above-circle learn gate
	- live smoke still needs fixture-state management, especially around slot-capped spellbooks and location-sensitive rites
- What would be done differently next time:
	- front-load fixture-state normalization for smoke characters before the closeout matrix
	- treat command-surface boundary checks as first-class closeout gates, not optional confirmation
	- document above-circle learning examples from shipped registry data rather than from rough tier expectations
