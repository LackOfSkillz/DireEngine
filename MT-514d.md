# MT-514d — Render-time state diagnostic

## Background

The MT-514 arc has built four runtime state systems: calendar, weather,
invasion, and terrain. MT-514c-impl shipped catalog-driven foraging
that consumes all of them. The runtime state model is now visibly
working in gameplay — Jekar foraged in `#4222 Kingshade Street`
post-fix1 and got terrain-appropriate items.

But there's an open architectural question about whether room
descriptions actually reflect runtime state in the live game. The
invasion dispatch's Phase H surfaced this:

> The current room display path in `rooms_extended.py` selects
> pre-split `desc_<state>` attributes and room tags, and I did not
> find a Dragonsire runtime evaluator for literal `$state(...)`
> fragments.

That was reported as blocked at the time. The runtime evaluator may
or may not exist; the agent didn't find it during invasion's Phase H
scope. We've never explicitly verified.

This dispatch answers two related questions:

1. **`$state(...)` markup runtime evaluation:** Do room descriptions
   parse and evaluate `$state(weather, ...)`, `$state(invasion, ...)`,
   `$state(season, ...)` etc. fragments at render time? Or is the
   markup machinery purely a generation-time concept (the AI uses it
   to know what variants to write, but the live render path ignores
   the syntax)?

2. **`desc_<state>` attribute switching:** Does the room render path
   pick `desc_storm` over `desc_clear` when current weather state
   changes? More generally, when there are multiple `desc_<state>`
   variants on a room, does the engine select the right one at
   render time based on current state?

These are diagnostic questions, not implementation questions. The
output is a written assessment that informs whether description
rendering needs follow-up work, and what shape that work would take.

This dispatch is read-only on production code. It does NOT implement
runtime state markup support or fix gaps. Findings drive whatever
implementation dispatch comes next (if any).

## Architectural guardrails (READ FIRST)

This is a diagnostic dispatch. The temptation will be to "just fix
the obvious gap" once it's identified. Resist that. Diagnosis
produces evidence and recommendations; implementation is a separate
dispatch that the user authorizes after reviewing findings.

The second risk is scope drift across all of description rendering.
The dispatch focuses specifically on state-aware rendering — how
runtime state (weather, invasion, season, time-of-day, terrain) is
or isn't reflected in what players see when they `look` at a room.
Other description rendering concerns (formatting, exits, contents
listing, lighting effects, etc.) are out of scope.

**Frozen scope:**

1. Phase A: Read-only audit of the room render path. Identify
   where descriptions are produced, what attributes are read, what
   transformations apply. Document precisely.
2. Phase B: Audit `$state(...)` markup syntax presence in the
   codebase. Where does it appear (templates, prompts, content)?
   Where is it parsed? Is the parser actually invoked at render
   time?
3. Phase C: Audit `desc_<state>` attribute switching. Do rooms have
   multiple `desc_<state>` attributes? Does the render path select
   among them based on current runtime state? If so, which states?
4. Phase D: Live observation. Find a room with state-aware content
   (or contrive one in a test fixture). Observe what the player
   sees as runtime state changes. Confirm or refute the audit
   findings empirically.
5. Phase E: Produce a findings document at
   `docs/audits/render_time_state.md` covering what works, what
   doesn't, and recommendations for closing any gaps.
6. Phase F: Validation tracker at `exports/mt514d_validation.md`.

**Frozen what-not-to-do list:**

- DO NOT implement `$state(...)` parsing if missing. That's a
  separate dispatch authored from this one's findings.
- DO NOT add `desc_<state>` switching logic if missing. Same.
- DO NOT modify any room descriptions, zone YAMLs, or content
  files. The audit reads existing data; it doesn't author new.
- DO NOT modify the calendar, weather, invasion, terrain, foraging,
  or any other runtime state module.
- DO NOT modify room typeclasses or display logic.
- DO NOT modify state-markup templates or AI generation prompts.
- DO NOT modify tests; this dispatch is read-only on the test suite.
- DO NOT introduce new render-time helpers, parsers, or selectors.
- DO NOT speculate beyond what the codebase actually shows. If
  something is unclear or partial, document it as unclear rather
  than guessing.
- DO NOT propose unrelated improvements to description rendering
  (formatting, accessibility, mobile support, etc.). Stay focused
  on state-aware rendering specifically.

**Stop-and-report conditions:**

- If the audit reveals the description render path is more complex
  than expected (multiple competing render paths, complicated
  inheritance, state-aware logic in unexpected places), stop and
  report. We may need to scope the diagnostic more narrowly.
- If `$state(...)` parsing exists but is partial (handles some
  state groups but not others), document that precisely. Don't
  oversimplify findings.
- If `desc_<state>` switching exists but covers some states and not
  others, same — document precisely.
- If the live observation in Phase D reveals behavior the audit
  didn't predict, treat the live observation as authoritative and
  document the discrepancy. The audit's job is to predict what
  live behavior should be; if it fails, that's information.
- If finding a room with state-aware content requires building one
  from scratch, that's fine — make a minimal one in a test fixture
  for verification. Do not modify production zone content.
- If the findings reveal a fundamental architectural choice (e.g.,
  state-aware rendering was deliberately deferred and there's a
  design rationale somewhere), surface that rationale rather than
  treating the gap as an oversight.

## Phase A — Audit the room render path

The agent reads and documents:

### A.1 Entry points

Where does room description rendering start? Likely candidates:
- `typeclasses/rooms.py` — base room class
- `typeclasses/rooms_extended.py` — extended room class (mentioned in
  invasion dispatch findings)
- Evennia's `return_appearance()` or `get_display_desc()` overrides
- Custom `look` command implementations

Document the actual entry point(s) used in production.

### A.2 What attributes are read

For each entry point, document:
- Which `db.<attribute>` reads occur during rendering
- Whether there's any conditional logic based on runtime state
- Whether tags are consulted (`room.tags.get(...)`)
- Whether external systems are queried (`get_current_weather()`,
  `get_current_invasion()`, calendar APIs)

### A.3 Transformations applied

If the description text undergoes any transformations between
attribute read and player display:
- Variable substitution
- Markup parsing
- Markdown/style processing
- State-conditional rendering

Document each transformation. If none exist (the description is
just the raw `db.desc` attribute output verbatim), document that
clearly — that's a meaningful finding.

### A.4 Multiple description fields

Document whether rooms support multiple description attributes:
- `db.desc` — base description
- `db.desc_winter` / `db.desc_summer` etc. — seasonal variants
- `db.desc_day` / `db.desc_night` — time variants
- `db.desc_<weather>` — weather variants
- `db.desc_<state>` — generic state variants

If multiple `desc_<state>` attributes exist on rooms, document:
- Which states are covered (audit a sample of rooms across zones)
- Whether the render path actually selects among them
- The selection logic, if present

## Phase B — `$state(...)` markup audit

The agent searches the codebase for `$state(` syntax and documents:

### B.1 Where the syntax appears

```bash
grep -rn "\$state(" world/builder/templates/
grep -rn "\$state(" world/builder/prompting/
grep -rn "\$state(" world/worlddata/
grep -rn "\$state(" docs/
```

Catalog every occurrence: file, line, the actual syntax used,
context.

### B.2 Where the syntax is parsed (or not)

Search for parsers, evaluators, or interpreters that handle the
syntax:

```bash
grep -rn "state(" --include="*.py" | grep -i "parse\|evaluate\|render"
grep -rn "_state_pattern\|STATE_PATTERN\|state_re" --include="*.py"
grep -rn "re.sub.*state\|re.findall.*state" --include="*.py"
```

If no parser is found, document that explicitly. Absence of
runtime parsing means `$state(...)` syntax in description text would
appear *literally* to the player rather than being evaluated.

### B.3 Test coverage

Are there tests that verify `$state(...)` evaluation at render time?
If yes, what do they assert? If no, that's a meaningful gap.

### B.4 Generation-time vs render-time distinction

Determine whether `$state(...)` syntax is:
- **Generation-time only:** AI uses it as a planning aid; the
  resulting room description is pre-split into `desc_<state>`
  attributes; the literal `$state(...)` syntax never reaches a
  rendered room description.
- **Render-time:** The literal syntax persists in `db.desc` and
  is parsed when the player looks at the room.
- **Hybrid:** Some descriptions are pre-split, others use literal
  syntax that's parsed at render time.

This distinction is the heart of the diagnostic. The answer drives
everything else.

## Phase C — `desc_<state>` attribute switching audit

If Phase A revealed the codebase supports multiple `desc_<state>`
attributes on rooms, dig deeper:

### C.1 Coverage in actual content

Sample existing rooms across zones. How many have multiple
`desc_<state>` attributes? Which states are covered? Document the
coverage rate.

For example:
- Rooms with `desc_clear` AND `desc_storm`: N
- Rooms with `desc_day` AND `desc_night`: M
- Rooms with `desc_invaded`: K
- Rooms with only `desc` (no state variants): the rest

### C.2 Selection logic

If the render path selects among `desc_<state>` attributes:
- What's the priority order when multiple states apply? (e.g., is
  `desc_invaded` preferred over `desc_storm` if both exist and
  both states are active?)
- What's the fallback when the current state has no matching
  variant? (Falls back to `desc`?)
- Is the selection cached or computed every render?

### C.3 Test coverage

Are there tests that verify `desc_<state>` switching? What do they
assert?

## Phase D — Live observation

The agent connects to the live server and observes whether the
audit's predictions match reality.

### D.1 Test scenarios to observe

Based on Phase A-C findings, design observation scenarios that
directly test the predictions. Examples:

- **If `$state(...)` is supposed to work:** Find or contrive a room
  with literal `$state(...)` syntax in its description. Look at it
  with different runtime states active. Confirm whether the syntax
  is evaluated or appears literally.

- **If `desc_<state>` switching is supposed to work:** Find or
  contrive a room with `desc_clear` and `desc_storm`. Set zone
  weather to clear, look. Set to storm, look. Confirm whether the
  description changes.

- **If neither works:** Confirm by looking at any room with state-
  aware content; the description should be the static `desc`
  regardless of state.

### D.2 Use a real character in real rooms when possible

Prefer Jekar in `#4222` or other live rooms over test fixtures.
Test fixtures don't represent production reality (we learned this
hard in the weather perf arc). Modify state via admin commands
(`@weather`, `@invasion`) and observe the room description through
`look`.

If suitable rooms don't exist in production, build a minimal
test-fixture room with state-aware content. Document that the
fixture was needed.

### D.3 Capture verbatim outputs

For each observation:
- The exact `look` output before state change
- The state-modifying admin command issued
- The exact `look` output after state change
- Any differences

This is evidence, not impression. Verbatim captures.
