# MT-516 — Targeting, stackables, and display aggregation

## Background

MT-514c-impl shipped catalog-driven foraging that produces multiple
items per attempt. In live playtesting, the user immediately hit the
limits of the existing object presentation:

```
> forage
You recover useful leaf, useful leaf.
> app lea
More than one match for 'lea' (please narrow target):
 useful leaf-1 (carried)
 useful leaf-2 (carried)
```

Two fungible items in inventory, no stack semantics, ugly disambiguation
suffix. The `leaf-1 / leaf-2` syntax is breaking immersion.

The same problem appears in two other directions:

- **Room descriptions:** "You also see a dagger, a dagger, a dagger,
  a dagger, ..." for 15 daggers. Bad UX regardless of whether the
  daggers are storage-stackable.
- **Targeting non-stackable distinct items:** Three goblin corpses
  in a room. `loot corpse` should auto-target sensibly. When the
  player wants a specific one, ordinal positional language should
  work — `loot first corpse`, `loot second corpse`, etc., scaling to
  any reasonable number.

This dispatch implements three intertwined things:

1. **Storage stacking for fungibles** — leaves stack into one entity
   with a quantity attribute; player sees "leaves (15)" not 15 entries
2. **Display aggregation** — anywhere the engine lists objects to the
   player (room descriptions, inventory, container contents),
   identical-named visible objects aggregate into counts. "15 daggers"
   not "a dagger, a dagger, ..."
3. **Targeting refinement** — auto-target on most-recent for the
   common case, three equivalent positional syntaxes (English
   ordinals, numeric-suffix ordinals, numeric positional) for explicit
   targeting, eliminate the `target-N` disambiguation suffix entirely

These are three problems with one underlying root: the engine treats
identical-looking objects as a list of separate things to disambiguate
between, when it should treat them as a count of fungibles or as a
naturally-orderable collection.

The user has shared a target resolver implementation from a parallel
DR-style Evennia project (Slippy's `TargetResolverMixin`, preserved
at `docs/references/slippy_target_resolver_2026-05-01.md`). That code
is architectural reference, not a drop-in module. The DireEngine
implementation is independently authored but adapts proven patterns
from the reference.

User decisions locked for this dispatch:

1. **Storage stacking for fungibles only.** Items flagged stackable
   merge into a single object with a quantity attribute. Items not
   flagged stackable remain individual entities (daggers, corpses,
   custom items with per-instance state).

2. **Identity-based stacking, not category-based.** Two "useful leaf"
   objects stack. A "useful leaf" and a "perfect leaf" don't (different
   identity). Quality-preserving category stacks deferred to v2 if
   identity-based proves too granular.

3. **Display aggregation broader than storage stacking.** Identical-
   visible-name objects aggregate in any listing surface (room desc,
   inventory, container contents) regardless of whether they're
   storage-stacked. So 15 separate dagger entities in a room display
   as "15 daggers" without being merged at the database layer.

4. **Aggregation key is the player-visible display string.** "a sharp
   dagger" and "a dagger" stay separate counts (different display
   strings). Two "a dagger"s aggregate.

5. **Exact counts in v1.** "15 daggers", "311 flowers".
   Narrative phrasing variations ("dozens of daggers", "a pile of flowers")
   deferred. Use natural English for one ("a dagger") and exact count
   for multiples ("2 daggers", "15 daggers").

6. **Auto-target on most-recent arrival** for non-stackable disambig.
   `loot corpse` with three corpses targets the most-recently-arrived
   match. Player doesn't need to specify in the common case.

7. **Three equivalent positional syntaxes for explicit targeting:**
   - English ordinal: `first`, `thirteenth`, `three hundred eleventh`
     — full English support, no upper cap
   - Numeric-suffix ordinal: `1st`, `13th`, `311th` — handles
     `1st/2nd/3rd` irregulars correctly
   - Numeric positional: `1.target`, `13.target`, `311.target` —
     compact form for power users and scripts

   All three resolve to the same target. Player picks what feels
   natural in the moment.

8. **Eliminate the `target-N` suffix entirely.** No more `leaf-1`,
   `corpse-2`. Display aggregation handles the listing layer; ordinal
   targeting handles the explicit-pick layer; auto-target handles the
   default.

9. **Apply consistently across all object-targeting commands AND all
   object-listing surfaces.** This is not a per-command feature; it's
   a project-wide change to how the engine presents and resolves
   object identity to players.

## Architectural guardrails (READ FIRST)

This is the largest dispatch since MT-514c-impl. It touches three
intertwined concerns and many code surfaces. The biggest risks:

- **Scope drift.** Touching object presentation will reveal adjacent
  improvements (item descriptions, examine output, look filters).
  Resist. This dispatch is targeting, stackables, and aggregation.
  Other improvements queue separately.

- **Breaking existing commands.** Many commands target objects.
  Refactoring resolution centrally must not silently change behavior
  for commands that worked correctly before. Existing test coverage
  is the regression guard.

- **Mismatched stacking and aggregation.** Storage stacking and
  display aggregation are different concerns. Conflating them in
  implementation produces bugs (e.g., trying to "split a stack" when
  the items are actually individual entities, or failing to aggregate
  individual entities because they're not stacks).

**Frozen scope:**

1. Phase A: Read Slippy's reference implementation at
   `docs/references/slippy_target_resolver_2026-05-01.md` (or wherever
   the user has saved it). Extract patterns to adapt; identify what
   doesn't fit DireEngine's data model. Document in validation artifact.
2. Phase B: Build the ordinal helper module at
   `world/helpers/ordinals.py` (or equivalent). Three syntaxes parsed
   to integers: English ordinals, numeric-suffix ordinals, numeric
   positional. Pure module, no dependencies on game logic.
3. Phase C: Build the target resolver mixin (or service) at
   `world/helpers/target_resolver.py`. Adapts Slippy's pattern with
   DireEngine's scope vocabulary. Centralized resolution that all
   object-targeting commands consume.
4. Phase D: Build the display aggregation helper at
   `world/helpers/display_aggregation.py`. Used by any code that
   lists objects to the player. Groups by player-visible display
   string, produces "N item" or "a item" output.
5. Phase E: Storage stacking infrastructure. Add a `stackable` flag
   to item typeclasses (or item metadata). Implement merge-on-receive
   for stackable items in inventory. Implement split semantics
   (`drop 5 leaves` from a stack of 20).
6. Phase F: Wire display aggregation into room descriptions. The
   "you also see" list (or equivalent) groups identical-display-name
   objects. Existing room rendering path consumes the new helper.
7. Phase G: Wire display aggregation into inventory listing. Same
   helper, different surface. Stacks display as their natural form
   ("leaves (15)"); non-stacked aggregates display as counts
   ("3 daggers").
8. Phase H: Wire target resolver into existing object-targeting
   commands. Identify all commands that resolve named targets;
   migrate each to the centralized resolver. Verify behavior
   preservation.
9. Phase I: Eliminate the `target-N` suffix from output. The default
   Evennia disambiguation prompt that produces `leaf-1, leaf-2`
   should not fire — auto-target or ordinal resolution handles
   ambiguous cases now.
10. Phase J: Tests at `tests/test_targeting.py`, `tests/test_ordinals.py`,
    `tests/test_display_aggregation.py`, `tests/test_stackables.py`.
    Focused unit tests parallel to existing patterns.
11. Phase K: Live verification with fixtures. Forage many leaves,
    check inventory display. Spawn many daggers in a room, check
    room display. Test all three ordinal syntaxes against fixtures.
12. Phase L: Documentation at `docs/architecture/object_presentation.md`.
    Defines stackability, aggregation rules, targeting syntaxes,
    authoring guidance for content creators.
13. Phase M: Validation artifact at `exports/mt516_validation.md`.

**Frozen what-not-to-do list:**

- DO NOT extend stacking to category-based merging (different
  qualities of leaf merging into a single stack with quality
  metadata). Locked decision: identity-based only for v1.
- DO NOT add narrative phrasing variations ("dozens of", "a pile
  of"). Locked decision: exact counts only for v1.
- DO NOT add a COLLECT command. The dispatch is about presenting
  what foraging already produces, not adding new gathering verbs.
  COLLECT remains queued for a future dispatch.
- DO NOT modify the foraging system, weather, invasion, calendar,
  terrain, or any runtime state module. This dispatch consumes the
  objects those systems produce; it doesn't modify the producers.
- DO NOT modify the state-aware description rendering (MT-514d-impl).
  Display aggregation operates on the room contents listing, not on
  description prose.
- DO NOT add admin commands or tooling for stackables. No
  `@stack`, no `@unstack`, no `@aggregate`. Stacking is automatic
  for stackable items; aggregation is automatic in listing surfaces.
- DO NOT extend the ordinal parser beyond the three locked syntaxes.
  No "the second one" natural language ("loot the second one").
  No relative ordinals ("loot last corpse" is out of scope for v1).
- DO NOT modify object descriptions, names, or display strings.
  Aggregation reads the existing display string; it doesn't author
  new ones.
- DO NOT modify NPC or character presentation. The dispatch is
  about objects (items, props, corpses). NPC/character listing has
  its own conventions and stays unchanged.
- DO NOT introduce new typeclasses beyond what's strictly needed.
  Stackable is a flag/attribute, not a new typeclass hierarchy.
- DO NOT modify item creation paths beyond the merge-on-receive
  hook for stackables. Foraging, inventory transfers, drops, gives,
  loots — all reuse existing creation paths and let the stack-merge
  logic happen at the inventory boundary.
- DO NOT pre-aggregate at the database layer. Display aggregation
  is render-time, not storage-time. Daggers stay as separate
  database objects; only the display groups them.
- DO NOT change Evennia's underlying object identity or dbref
  semantics. Stackables are still individual database objects with
  a quantity attribute; they don't become a fundamentally new kind
  of database entity.

**Stop-and-report conditions:**

- If the existing object-targeting commands have inconsistent
  resolution patterns (some use Evennia's default search, some
  use custom matchers, some use heuristics), document the
  divergence and surface before centralizing. May need a smaller
  initial migration scope.
- If display aggregation conflicts with existing room rendering
  (e.g., the "you also see" list is itself state-aware via
  MT-514d-impl markup), stop and report. We may need a layered
  approach.
- If stackable merge-on-receive surfaces an existing bug (e.g.,
  inventory operations don't go through a clean boundary that
  hooks can attach to), stop and report.
- If the ordinal parser hits English-edge-case ambiguities that
  the locked decision didn't resolve (e.g., "first" vs "1st" vs
  "1." spacing/punctuation handling), document the question and
  pick a defensible default; flag in the validation artifact.
- If migrating commands to the centralized resolver breaks
  existing tests in ways that aren't attributable to correct
  behavioral change, stop and report.
- If eliminating the `target-N` suffix breaks scripts or admin
  tooling that depended on it, stop and report. The suffix may
  need a deprecation path rather than immediate removal.
- If the bounded-time test reveals targeting/aggregation is slow
  (a `look` at a room with many objects taking more than a few
  ms), stop and report.
- If live verification reveals output the unit tests didn't predict,
  treat live as authoritative and stop and report.

```python
def parse_numeric_suffix_ordinal(text: str) -> int | None:
    """Parse '1st', '2nd', '3rd', '13th', '311th'.
    Validates suffix matches number (e.g., '2nd' valid, '2st' not)."""

def parse_numeric_positional(text: str) -> int | None:
    """Parse '1.target' or 'target.1' (agent picks one canonical form).
    Returns the integer; rest of phrase consumed elsewhere."""
```

### B.2 The combined parser

A higher-level function that accepts any phrase and identifies
which form is being used, then returns the integer plus the
remaining phrase (the noun being targeted).

```python
def parse_ordinal_and_name(phrase: str) -> tuple[int, str]:
    """Returns (n, name).
    n=1 if no ordinal present.
    name is the phrase with ordinal stripped."""
```

### B.3 The "other" alias

`other dagger` resolves to ordinal=2. Documented in the helper.

### B.4 Tests

`tests/test_ordinals.py`:

- "first" through "tenth" — common cases

## Phase C — Target resolver

`world/helpers/target_resolver.py` (or wherever Phase A determined).

### C.1 Architecture

Mixin pattern, per Slippy's reference, adapted to DireEngine:

```python
class TargetResolverMixin:
    DEFAULT_SCOPES = (...)  # DireEngine's scope vocabulary

    def resolve_target(
        self,
        phrase: str,
        *,
        scopes: Iterable[str] | None = None,
        require_container: bool = False,
        possessive: bool | None = None,
    ) -> tuple[Object | None, int, str, str]:
        """Returns (resolved_object, ordinal_n, base_name, scope_name)."""
```

### C.2 DireEngine scope vocabulary

The agent inspects DireEngine's character data model and identifies
the scopes that fit:

- `inventory` — items the character carries
- `room` — objects in the character's location
- Possibly `worn` — if DireEngine has equipped slots; otherwise omit
- Possibly `hands` — if DireEngine has hand-tracking; otherwise omit

The exact scope set is what DireEngine actually supports, not what
Slippy used.

### C.3 Resolution priority

Per locked design:

1. Filter candidates by name match (token-prefix with noun safety)
2. Group candidates by scope
3. Sort within scope by creation time, newest-first
4. Apply ordinal: ordinal=1 → first match, ordinal=N → Nth match
5. Return the resolved object, plus metadata about how it resolved

### C.4 Auto-target default

When no ordinal is specified (`loot corpse`), ordinal defaults to 1.
The newest-first sort means "the most-recently-arrived corpse" is
the default match. This matches the locked auto-target rule.

### C.5 Stackable handling

When the resolved object is a stackable, the resolver returns the
stack object (single database entity with quantity). The caller
decides whether to operate on the whole stack or split.

For phrases like `drop 5 leaves`, the agent designs the parser to
extract the quantity component; the resolver returns the stack;
the calling command (drop) handles the split.

### C.6 Tests

`tests/test_targeting.py`:

- Single object in scope, name match → returns object
- Multiple objects, no ordinal → returns newest
- Multiple objects, ordinal=2 → returns second-newest
- Three ordinal syntaxes resolve to same target
- Non-existent target → returns None
- "other" alias resolves to ordinal=2
- Possessive "my X" filters to owned scopes
- Embedded container "X in Y" resolves nested

## Phase D — Display aggregation helper

`world/helpers/display_aggregation.py`.

### D.1 The function

```python
def aggregate_for_display(
    objects: Iterable[Object],
    *,
    looker: Object | None = None,
) -> list[tuple[str, int]]:
    """Group objects by player-visible display string.

    Returns list of (display_string, count) tuples preserving
    encounter order.

    Used by anywhere the engine lists objects to the player."""
```

### D.2 Display string resolution

The aggregation key is what the player sees, not the object's
internal name. For most items this is `obj.get_display_name(looker)`.
For stackable items already merged at the database layer, this is
the stack's display name (which already incorporates count).

### D.3 Output formatting

A separate function formats the aggregated list as natural prose:

```python
def format_aggregated_list(
    aggregated: list[tuple[str, int]],
    *,
    list_separator: str = ", ",
    final_separator: str = ", and ",
) -> str:
    """Format aggregated counts as 'a dagger, 3 swords, and 15 leaves'.

    For count=1: 'a/an X' (preserve existing article handling).
    For count>1: 'N Xs' (with simple pluralization)."""
```

### D.4 Pluralization

Simple English pluralization for counts > 1:
- Most words: append "s"
- Words ending in s/x/z/ch/sh: append "es"
- Special cases as needed (the agent uses a small lookup or library)

If pluralization is wrong for an item, content authors can override
via an explicit plural attribute on the item. v1 uses simple rules
and an override hook.

### D.5 Tests

`tests/test_display_aggregation.py`:

- Single object → "a dagger"
- Two identical → "2 daggers"
- Three different → "a dagger, a sword, and a shield"
- Mixed identical and different → "2 daggers and a sword"

## Phase E — Storage stacking

### E.1 Stackable flag

Items declare stackability via:
- Item typeclass property: `is_stackable = True`
- OR item attribute: `db.stackable = True`

Agent picks based on existing typeclass conventions. The flag is
declarative; the merge logic consumes it.

### E.2 Forage catalog integration

The forage catalog at `world/builder/content/forage_catalog.yaml`
already has 139 items. The agent identifies which categories should
be stackable:

- `category: filler` (grass, sticks) — yes, stackable
- `category: healing_herb` (most leaves) — probably yes for v1, even
  though quality nuance exists. Identity-based stacking means
  different healing herbs are different stacks; same identity merges.
- `category: rare_herb` — yes, stackable, same logic
- `category: edible` (berries, mushrooms) — yes
- `category: utility` (tinder, charms) — case-by-case

The agent picks reasonable defaults per category and documents.
Content authors can override per-item later if needed.

### E.3 Merge-on-receive

When a stackable item enters an inventory:
1. Check if a matching stack already exists in the inventory
   (matching by item identity, not by object identity)
2. If yes: increment the existing stack's quantity, delete the
   incoming object
3. If no: keep the incoming object as-is (it becomes the new stack)

Identity matching: same typeclass + same key + same display
attributes. Agent designs the comparison; documents.

### E.4 Display name with quantity

Stack display name includes quantity:
- "leaves (15)" or "15 leaves" — agent picks based on existing
  conventions; documents

### E.5 Split semantics

When the player operates on a partial stack:
- `drop 5 leaves` from a stack of 20: stack quantity decreases to 15,
  a new stack of 5 is created and dropped
- `drop leaf` (no count): drops 1 from the stack
- `drop all leaves`: drops the whole stack

Split logic in inventory operations: drop, give, put.

### E.6 Tests

`tests/test_stackables.py`:

- Two identical stackable items merge on entering inventory
- Different identity items don't merge
- Non-stackable items don't merge regardless of identity
- Drop 1 from stack: stack quantity decreases, new single-item stack
  created
- Drop N from stack: stack quantity decreases by N, new N-stack
- Drop all: full stack moves
- Drop more than stack: error or drops all (agent picks; documents)
- Display name reflects current quantity
